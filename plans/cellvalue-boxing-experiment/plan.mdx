# Experimento: `CellValue` struct vs. `object?`

Provar, com BenchmarkDotNet e em escala pequena, se trocar o retorno de `Expression.Compute` de `object?`
por um value type opaco `CellValue` elimina o boxing do caminho numérico **e** melhora o throughput o
suficiente para justificar a migração completa do avaliador. O experimento roda na branch
`experiment/cellvalue-struct` como um spike isolado no projeto de benchmark — **nenhum** dos 64 overrides
de produção é tocado até o número decidir.

**Done** = a tabela do BenchmarkDotNet compara as duas variantes nos mesmos workloads, e existe um veredito
explícito (migrar / plano-B / abandonar) escrito contra um gate definido de antemão.

**Fora de escopo:** migrar os 64 `Compute` reais, mexer no `Workbook._cache`, ou tocar em `ValueCoercion`
de produção. Isso só acontece num plano futuro, se o gate for atingido.

### Por que o boxing é o alvo

Hoje `Expression.Compute(EvaluationContext) : object?` (`Danfma.MySheet/Expressions/Expression.cs:75`). O
plano de memoização já mediu que, depois de `EvaluationContext` virar `readonly struct`, sobra **um único
alocador no caminho quente: o box do `double`** (24 B por resultado). Cada célula numérica computada aloca
um box de curta vida que morre em Gen0. Em extração em lote de milhares de células, isso vira pressão de GC
real. É esse box que o experimento tenta matar.

<Callout id="mechanism-decision" tone="decision">

**Mecanismo travado: `readonly struct` de dois campos + tag — sem `Unsafe`.** A ideia original de emular
uma union sobrepondo `double` e uma referência (`IntPtr`) via `[StructLayout(Explicit)]`/`Unsafe` **não
carrega no CLR**: o GC é preciso e proíbe sobrepor um campo gerenciado (`object`/`string`) com um
não-gerenciado no mesmo offset (`TypeLoadException` na carga do tipo). Guardar objeto como `IntPtr` cru
exigiria `GCHandle`/pinning — mais caro que o box que se quer evitar. C# 14 / .NET 10 ainda não têm
discriminated unions nativas, então emula-se com struct + tag. Nada de `Unsafe`, `FieldOffset`, `IntPtr`.

</Callout>

O ponto elegante: Number, Bool, Blank e Error cabem todos no campo `double` + tag, então o caminho quente
fica **livre de alocação** — inclusive os erros deixam de ser objetos no heap (o código do erro vai como
`int` no `_num`). Só Text/Reference tocam o campo `object?`, e mesmo esses guardam uma referência que já
existe, sem alocar nada novo.

### Esboço do `CellValue`

<Code
  id="cellvalue-sketch"
  filename="benchmarks/Danfma.MySheet.Benchmark/Spike/CellValue.cs"
  language="csharp"
  caption="Struct de dois campos + tag. Sem Unsafe. AsObject() é a ponte para os call sites antigos durante uma futura migração."
  code={`public enum CellValueKind : byte { Blank, Number, Boolean, Error, Text, Reference }

public readonly struct CellValue
{
    private readonly double _num;    // Number | Boolean(0/1) | Error(code) | Blank
    private readonly object? _ref;    // Text(string) | Reference — null nos escalares
    private readonly CellValueKind _kind;

    private CellValue(double num, object? reference, CellValueKind kind)
        => (_num, _ref, _kind) = (num, reference, kind);

    public static readonly CellValue Blank = new(0, null, CellValueKind.Blank);
    public static CellValue Number(double n) => new(n, null, CellValueKind.Number);
    public static CellValue Bool(bool b) => new(b ? 1 : 0, null, CellValueKind.Boolean);
    public static CellValue Error(int code) => new(code, null, CellValueKind.Error);
    public static CellValue Text(string s) => new(0, s, CellValueKind.Text);

    public CellValueKind Kind => _kind;

    // Caminho quente: sem unbox, sem checagem de tipo por reflexão.
    public bool TryGetNumber(out double value)
    {
        value = _num;
        return _kind is CellValueKind.Number or CellValueKind.Boolean;
    }

    // Ponte para call sites que ainda esperam object? (migração gradual).
    public object? AsObject() => _kind switch
    {
        CellValueKind.Number => _num,
        CellValueKind.Boolean => _num != 0,
        CellValueKind.Blank => null,
        _ => _ref,
    };
}`}
/>

<Callout id="copy-risk" tone="risk">

**O que pode matar o ganho — e por isso medimos.** `CellValue` (~24 B) é 3× o tamanho de um `object?`
(8 B) para passar por valor num avaliador recursivo (105 call sites), acima do ponto-doce de 16 B onde o
JIT enregistra o struct com folga. Parte do ganho de não-alocar pode voltar como tráfego de cópia na pilha.
O saldo é *provavelmente* positivo, mas não é garantido — a decisão de migrar 64 arquivos tem que sair de
um número, não de convicção.

</Callout>

### Escopo do experimento

Spike isolado em `benchmarks/Danfma.MySheet.Benchmark`, com dois mini-ASTs equivalentes — uma variante
`object?` (baseline, espelhando o `ValueCoercion.TryToNumber` atual com unbox) e uma variante `CellValue`
(coerção via `TryGetNumber`, sem unbox). Modela o **domínio escalar completo** (Number/Bool/Blank/Error
inline; Text/Reference no campo `object?`), com cargas **dominadas por aritmética** para refletir o caso de
uso real de extração numérica.

## Phase 2b — Cenário fiel: grafo memoizado + cache `CellValue` · ✅ Concluída

O spike da Fase 2 isolava o box transitório num `Eval` puro — não cobria ranges, grafo de dependências com
caminhos cruzados/compartilhados (A←B←C, D←B), nem o **cache**. É no cache que o boxing mais pesa: hoje
`Workbook._cache` é `ConcurrentDictionary<…, object?>`, então cada valor numérico cacheado é um **box de vida
longa** (promove para Gen1/Gen2). Este experimento compara `Dictionary<string, object?>` vs
`Dictionary<string, CellValue>` numa extração em lote memoizada.

<Callout id="phase2b-reading" tone="risk">

**Dois achados — um corrige o veredito da Fase 2.** (1) **Alocação confirmada exatamente:** o cache `object?`
boxa **24,0 B/célula** (medido 23,99 B) de **vida longa** → dispara **Gen1** (171,9/1000 ops em 100k); o cache
`CellValue` aloca **0 B** e tem **zero** coletas (Gen0 e Gen1). Elimina a pressão de GC de geração alta — o que
mais importa num extrator de background. (2) **Throughput modesto aqui:** só **4–12%** mais rápido —
**abaixo dos 15%** — porque lookups de `Dictionary<string,…>` dominam esse caminho, não o box. Contrasta com
os 2,3×–4,3× do caminho aritmético puro.

</Callout>

## Phase 3 — Decisão contra o gate · ✅ Concluída (veredito: MIGRAR, por GC/memória)

Objetivo: comparar os números ao gate e escrever o veredito, sem tocar produção.

<Callout id="phase3-verdict" tone="decision">

**MIGRAR — com o argumento correto: memória/GC, não só velocidade.** `CellValue` zera a alocação em TODOS os
cenários, mas o throughput depende do caminho: **2,3×–4,3×** no eval-heavy (aritmética, gate passa folgado) vs
**4–12%** no cache-heavy (extração em lote de grafo — o caso de uso real declarado — **abaixo dos 15%**). A
justificativa forte e universal é a **eliminação de alocação + pressão de Gen1** num processo de longa duração
(alvo do memoization plan). O plano-B (box-cache) foi rejeitado; o risco da cópia de 24 B não se materializou.
Se a decisão pesar só throughput bruto no caminho cache-heavy, o gate não é cruzado; se pesar GC/latência de
cauda, o caso é forte. Migração dos 64 arquivos = plano separado, não executada aqui.

</Callout>
